### Q.

- 구글에 JWT관련 예제를 보면 즉시 로그아웃 문제, 보안문제, 기타 다른 문제들이 있어서 refresh token을 서버에서 redis 같은 곳에 관리하고 access token만 client에게 발급하는 방식이 많이 보이는데요.

- 이렇게 구현하면 세션과 마찬가지로 서버에서 refresh token을 관리하는거니까 stateful한거고 그럼 세션의 문제점이 따라올거같은데 이런 방식으로 JWT를 구현해도 JWT가 가지는 장점이 있나요??

- 그리고 이런 방식이 추천되는 방식이 맞는지 궁금합니다.

### A.

- 좋은 질문이예요 👍

- 우선 첫번째, Refresh Token (지금부터 편의상 RT라고 할게요)은 토큰을 강도당하는 보안 취약점을 전혀 개선해 주지 않아요. RT을 사용할때 클라이언트 측에서 안전한 방식으로 보관하도록 신경을 써줘야 해요. (JWT도 마찬가지!) 보통 RT는 AT(Access Token)보다 수명이 길기 때문에 (RT 특성상 만료 기간이 더 길어요), 한번 해커가 훔쳐가면 더 위험성이 크기 때문에 RT를 더 보안에 신경써줘야 하죠.

- 다만, 세션을 사용하면 서버에서 강제적으로 로그아웃을 시키는 부분을 구현할 수 있는데, JWT는 그렇게 하지 못하죠. RT는 이런 JWT의 단점을 개선할 수 있지만 앞에서도 언급했듯 보안 자체를 크게 개선해 주지는 못해요. (클라이언트 측에서 관리 잘못하면 더 망할..가능성이...ㅎㅎㅎ)

- 두번째로는, 많은 분들이 Redis나 다른 중앙 Auth관리 서비스를 사용하면 서비스를 stateful하게 만든다고 오해 하시는데, 상태(state)를 한 서버에만 로컬적으로 가지고 있는 경우에만 서비스가 stateful 하지, 중앙 관리형 Redis 같은 경우는 하나의 서버 로컬에 정보가 있는것이 아니기 때문에 stateful 하다고 볼 수 없어요.

- 마지막으로, RT를 함께 사용한다고 해도 여전히 JWT를 사용하는데 아래와 같이 큰 장점이 있어요.

- Redis나 다른 Auth 서비스에 통신 하지 않고도 JWT를 통해서 바로 인증할 수 있다. (부가적인 네트워크 요청이 필요 없으므로 성능이 좋다)

- RT만 Redis나 다른 Auth 서비스에 통신 하면 됨

- 즉, 그냥 세션을 사용하게 되면, 모든 API 요청에 대해서 Redis나 다른 서비스에 통신해서 사용자의 요청을 인증해야 하지만, RT + JWT를 사용하면 모든 API 요청은 JWT만 확인하면 되고, RT를 업데이트 또는 발행해야 할때만 다른 인증 서비스를 이용하면 되죠 😆

JWT를 RT없이 보안에 안전하게 만드는 다른 방식은 보너스 챕터 한번 보세요 :)

### Q.

- 저기 치명적인 보안편에서 "xss어택에 안전한 백엔드 업데이트" 강의에서 궁금한게 있는데 거기에는 discussion란이 없어서 여기에 질문드립니다.

- 9분30초쯔음부터 나오는 강의 내용이 좀 이해가 잘 안되는데

1. "브라우저 client는 쿠키에 자동으로 들어온다" 라고 나오는데 이게 json body부분에 쿠키가 들어온다는 건가요?

2. "그외는 쿠키 사용안하므로 헤더에 온다" 라고 나오는데 브라우저에서도 헤더 부분에 Authorization: Bearer .... 이렇게 오지 않나요??

3. "헤더에 토큰이없으면 브라우저 클라이언트" 이 부분도 2번과 마찬가지로 브라우저 client에서 요청이 올때 헤더부분이 있지 않나요??

4. 모바일 클라이언트에게 요청이 올때는 헤더가 없이 json body만 오는건가요?

### A.

- HTTP Only 옵션을 사용해서 쿠키 (쿠키도 헤더에 들어있는 데이터예요)를 서버에서 클라이언트에게 보내게 되면, 브라우저 클라이언트는 자동으로 서버에서 받은 HTTP Only 마크가 되어있는 쿠키의 데이터를 나중에 다른 요청을 할때 자동으로 헤더에 포함해서 요청을 보내요. 하지만 다른 모바일 클라이언트들은 이런 기능이 없답니다 :)

- 쿠키는 헤더에 있어요. Cookie라는 키에 여러가지 쿠키들이 ; 세미클론으로 구분되어서 값으러 지정되어져 있죠

```
Cookie: token=value; cache=value;

```

- 브라우저외 클라이언트들은 Cookie가 아니라 헤더에 있는 Authorization 에서 받아오겠죠?

```
Authorization: value
```

## 16.6 가입, 로그인 Rest APIs 구현하기 💡

### Q. jwt 토큰을 헤더가 아닌 바디에 넣으신 이유는 무엇인가요?

- 안녕하세요 엘리님! 컨트롤러 코드(signup, login 구현)에서 질문이 있어서 남깁니다.

- 이번 강의를 보기 전에, 저는 구글링을 통해 아래와 같이 토큰을 헤더에, username을 바디에 넣어서 response를 짰습니다.

'''

res.header("Jwt-Token", token);

return res.status(200).json(username)

'''

- 하지만 강의 코드에서는 jwt토큰을 만든 후 아래와 같이 토큰과 username을 함께 response 바디에 넣어서 클라이언트에게 보낸 것 같은데 헤더가 아닌 바디에 넣어서 보내는 이유가 있나요?

'''

res.status(201).json({ token, username })

'''

### A.

- 전형적으로 클라이언트가 서버에 데이터를 보낼때 서버에서 요구하는 내용들을 헤더에 포함해서 보내야 해요 (Authorization 토큰 같은).

- 서버가 클라이언트에게 앱 데이터를 보낼때는, 클라이언트는 서버에서 보낸 반응에서 HTTP Status와 Body 부분을 확인한답니다.

- 정리 하자면, 전형적으로 이 반응을 얼마동안 캐쉬에 저장하느냐등 전체적인 HTTP Flow 흐름에 대해서 명시할때는 서버에서 헤더에 Cache와 같은 내용을 지정할 수는 있지만, 앱의 관련된 데이터는 (payload) body에 넣어서 보내요 :)

### Q. 서버에서 토큰을 클라이언트에 발급 해 줄 때

- 쿠키를 사용할 때는 쿠키를 가로채지 못하게 http only 옵션을 사용한다

- 그러나 rest api의 클라이언트는 http 웹만 있는게 아니다.

- 서버에서 토큰을 클라이언트에 발급해줄 때 jwt 토큰을 cookie에 설정하는 경우를 많이 봤다.

- 이 쿠키를 가로채는 경우를 대비 해 보안을 위해 access token, refresh token을 사용한다.

- access token의 유효시간을 짧게 해 사용자의 access token 유효시간이 끝났을 때 refresh 토큰을 통해 다시 access 토큰을 발급 받는 원리이다.

- 이렇게 이해 했는데요!

- " 엘리쌤께서 이 강의 다른 질문에 앱관련 data를 서버가 클라이언트에 보내줄 때 body에 포함한다고 하셨는데 이는 rest api라서 그런건가요? "

- jwt 토큰을 사용하는 다른 코드들을 보았는데 서버가 res.cookie에 설정해 발급하더라구요.

- express의 res.cookie는 set-cookie 헤더를 설정하는 거잖아요.

- 만약 rest api가 아닌 일반적인 웹에서만 사용하는 서버일 경우 cookie에 토큰을 발급하나요?

### A.

- 쿠키는 (HTTP 반응 헤더에 들어 있는 정보) 브라우저에서 사용하는 것으로, 모바일 어플리케이션에서는 쿠키 보다는 바디에서 데이터를 주로 읽어 온답니다.

## 16.7 Auth 미들웨어 구현하기 💡

- https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/Authentication

- 엘리의 추가 노트:

- 세션과 다르게 JWT 토큰 자체로 유효성을 판단하는 장점을 극대화 하기 위해서,
  isAuth에서 사용자 DB에 사용자가 있는지 없는지 확인하는 과정은 생략 가능합니다 ✨

- 사용자의 유효성을 확인하는것이 필수적인 상황이라면, 필요한 API에 한해서 컨틀롤러에서 확인하거나, 꽤 많은 API에서 확인이 필요한경우 isAuth처럼 별도의 미들웨어를 만들어서 확인할수 있어요.

- 그럼 어플리케이션 전반적으로 다 사용하는 isAuth 미들웨어에서는 JWT 토큰 자체로 유효성을 확인 할 수 있으니, (다른 DB의 쿼리를 할 필요가 없음) 더 좋은 성능을 기대해 볼 수 있겠죠? :)

### Q. auth/me 질문

- auth/me 라우터에서 하는 일이 client가 가지고 있는 토큰의 userId 정보를 가져와서 해당 유저가 서버에 있는지 확인하는 일로 이해를 하고 있습니다. 그런데 실질적으로 지금 이 프로젝트에서 어디서 쓰이는지를 모르겠어서 필요성이 사실 잘 와닿지가 않습니다. 구체적으로 이 프로젝트에서 어떤식으로 auth/me가 쓰이는지 알고 싶습니다! Auth 부분을 다 듣고 다시 들으며 생긴 궁금증입니다.

### A.

- 좋은 질문이예요 :) auth/me는 클라이언트단에서, 예를 들어 사용자가 dwitter 사이트에 로그인 했다가 다시 새로고침을 하거나, 다른 페이지로 이동했다가 다시 돌아오는 경우! 매번 로그인 해라고 로그인 페이지로 가면 사용자가 정말~~ 귀찮겠죠?

- 그럴때 auth/me API를 이용해서 이 사용자가 유효한지, 토큰이 만료되지는 않았는지 확인할 수 있어요.

- Dwitter 클라이언트인 리액트 앱에서 scr/context/AuthContext에서 아래와 같이 사용하고 있는걸 볼 수 있을거예요. 사용자가 앱을 열기만 하면 실행되는 코드로, 항상 사용자의 정보를 먼저 받아 와서, 사용자 정보가 있다면 사용자를 설정하고, 그렇지 않다면 로그인 페이지로 이동하겠죵?

```tsx
useEffect(() => {
  authService.me().then(setUser).catch(console.error);
}, [authService]);
```

### Q. isAuth 미들웨어에서 해독한 페이로드의 유저id로 데이터베이스 조회부분 여쭙니다!

- 안녕하세요 앨리, 토큰 확인 미들웨어에서 디비 조회하는 부분에 대해 문의드립니다!

- 저도 프로젝트 진행 시, 이렇게 Auth 미들웨어로 토큰 유효성 검사 후 비즈니스 로직으로 넘어가도록 구현을 했는데요

- 추후 세션과 토큰을 자세히 공부하면서, 또 앞 전 강의를 통해 토큰의 장점 중 하나가 바로 네트워크 비용 절감이라는 것을 알게됐습니다!

- 세션의 경우는 매 요청마다 세션DB에 해당 SID가 있는지 조회하지만 토큰은 토큰이 유효하다면 토큰 유무만으로 정상적인 사용자라 인식해야 한다고 습득했는데요!

- 그래서 기존 앨리님과 같은 로직에서 decoded한 userId로 DB에 실 사용자인지 확인하는 부분을 지우고 유효성 검사만으로 next()한 후에 해당 비즈니스 로직에서 userInfo가 필요할 때만 조회하도록 구현했습니다. 그러면 조회 비용이 조금 절감되더라구요!

- 혹시 그래도 Auth미들웨어에서 사용자를 확인하고 req 객체에 해당 정보를 넣어 넘겨주는 방식이 더 이로운지 문의드립니다. 항상 감사드립니다!

### A.

1. 넵, JWT의 유효성 검사만 해주어도 충분합니다👍 그리고 말씀하신것처럼 이게 JWT를 사용하는 장점중 하나 이기도 하죠! (별도의 세션 확인이 필요없음)

2. 하지만, 15.4 강의에서도 다루었듯, JWT는 한번 발행이 되면 만료될때까지(expired) 영원히 유효 하다는 단점이 있어요.

세션과 다르게, 백엔드에서 강제로 리셋할 수 있는 방법이 없죠.

예: 만약 A라는 사용자가 JWT를 한번 획득하고 나면, 토큰 자체가 만료될때까지, 계속 토큰을 사용할 수 있어요. (삭제된 사용자라도 토큰을 가지고 있으면 데이터를 읽고, 쓰고, 업데이트 및 삭제할 수 있음)

그래서 미들웨어에서 이 사용자가 이런 권한이 있는지 확인하는 절차가 필요해요.

3. isAuth 미들웨어에서 비용절감을 위해 JWT 유효성만 확인하고 싶다면: 대안적으로, 필요한 Controller에서만 사용자의 유효성을 확인해도 되고, 또는 사용자의 적합성을 확인하는 미들웨어를 따로 만들어서 필요한 곳에서만 그 미들웨어를 사용하도록 만들수 있어요.

(이 방법이 제일 좋은것 같아요 💯)

4. 우리 프로젝트,백엔드 어플리케이션에서 어떤 점을 더 중요하게 생각하는지 판단해서 그에 맞게 선택해서 구현하는게 제일 좋다고 생각해요 :)
